# isekai-ssh リモート発ビルドトリガー

## 前提となる状況

自分(Claude)は今、`isekai-ssh` 経由のリモートSSHセッションの中(=接続先ホスト)で作業して
いる。ユーザーが「クライアント側(`isekai-ssh` を実際に叩いているマシン、Windowsなど)で
しかビルド・実行確認できないもの」を試したがっている場合にこのスキルを使う。設計の詳細は
`ISEKAI_PIPE_DESIGN.md` §8 Epic P を参照(実装前に必ず目を通すこと、特に「セキュリティ」の
節)。

**このスキルが実現する処理の中身は一切ここでは決めない**。実行されるディレクトリ・
コマンドは常にクライアント側のローカル設定(`~/.config/isekai-ssh/build_profiles.toml`)が
握っており、リモート(=このセッション)からはプロファイル**名**しか送れない。このスキルは
その登録・呼び出しの手続きを手伝うだけ。

## 前提確認

1. `$ISEKAI_CTL_SOCK` が設定されているか確認する(`echo "$ISEKAI_CTL_SOCK"`)。空なら、
   クライアント側の `~/.ssh/config` に接続先ホストへの `#@isekai ctl-socket yes` が
   無いということなので、ユーザーにその旨を伝えて追記・再接続してもらう。ここで止まる。
2. 接続先ホストのエイリアス(ユーザーがクライアント側の `~/.ssh/config` で使っている
   `Host` 名)をユーザーに確認する。分からなければ聞く — 推測しない(build profileは
   host名で紐付いているため、間違えると「プロファイルが見つからない」で失敗するだけで
   実害はないが、聞いた方が早い)。

## 手順

### 1. プロファイル内容を決める

ユーザーから次を確認する(既に会話の中で分かっていれば聞き直さない):
- プロファイル名(例: `win`)
- クライアント側のビルド対象ディレクトリ(絶対パス)
- 実行するビルドコマンド(シェル文字列、`&&` 等も可——クライアント側のローカルシェルで
  そのまま実行される)
- (任意)成果物を回収したい場合: `--result-glob`(ビルドディレクトリ基準のglob)と
  `--dest-dir`(このリモートホスト上の送信先ディレクトリ)。片方だけの指定は
  `build-profile add` 自体がエラーにするので、両方セットか両方省略のどちらか。

### 2. 登録コマンドを組み立ててクリップボードへ送る

```bash
CMD="isekai-ssh build-profile add <host> <name> --dir <dir> --command <command> [--result-glob <glob> --dest-dir <dest>]"
printf '%s' "$CMD" | isekai-pipe ctl clip push --mime text/plain
```

送ったら、ユーザーに次のように伝える(要約せず、実際に何が起きたかを明確に):

> `<CMD>` をクリップボードに送りました。ローカルの端末に貼り付けて実行し、完了したら
> 教えてください。

**ここが唯一の人間判断ステップであり、絶対に自動化・省略してはいけない**。クリップボードに
乗ったコマンドをユーザーの代わりに実行したことにする、実行結果を推測する、確認を待たずに
次のステップへ進む、のいずれも禁止(`ISEKAI_PIPE_DESIGN.md` Epic M/Epic Pが明記する
「push側はクリップボードハイジャックの踏み台になりうる」というリスクそのものであり、
ここを自動化すると実質的に汎用リモートexecになってしまう)。

### 3. ユーザーの確認を待ってからビルドを実行する

ユーザーから「登録した」「実行した」等の確認が来たら、次を実行する:

```bash
isekai-pipe ctl build <name>
```

これはビルドが終わるまでブロックし、stdout/stderrをそのまま自分の標準出力に流す
(=ユーザーにもそのまま見える)。終了コードがビルド自体の終了コードになる。

同じ`isekai-ssh`セッションを再接続する必要は無い——プロファイルは`isekai-ssh`側で
ホットリロードされる(再起動・再接続なしで、直前に登録したプロファイルがそのまま使える)。

### 4. 結果を報告する

- 失敗した場合、`isekai-pipe ctl build`の出力に「no build profile registered」が
  含まれていれば、host名かプロファイル名の登録内容の食い違いを疑う(手順1に戻って
  ユーザーと一緒に確認する)。
- `--result-glob`/`--dest-dir`を設定していた場合、成果物の送信はバックグラウンドで
  行われ、このコマンドの終了を待っても届いているとは限らない(既知の制限、
  `ISEKAI_PIPE_DESIGN.md` §8 Epic P参照)。届いているかは`--dest-dir`を`ls`して
  確認する。
